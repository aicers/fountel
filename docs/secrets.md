# Secrets: local generation for dev (nothing committed)

fountel keeps **no secret material in the repository** — not even encrypted.
Dev secrets are random and disposable, so the dev stack **generates its own
secrets locally at bring-up** and writes them to a git-ignored directory. There
is no sops/age dependency and no out-of-band decryption key to hold or lose.

> **Why local generation (revised 2026-06-14, see #1).** Dev secrets are random
> and disposable, and fountel's feed integrity/authenticity rides on **mTLS +
> content hashes, not GPG-signature pinning**, so a reproducible instance
> signing key is not needed in dev. There is therefore nothing worth committing
> — committing encrypted secrets only added an age key that one operator had to
> hold (and lost once). prod will use a **secret manager** (not sops); that is
> Phase 2 and out of scope here.

## What is a secret here

| Secret | Used by | Notes |
| --- | --- | --- |
| `MYSQL_PASSWORD` | misp-core, db | App DB user. |
| `MYSQL_ROOT_PASSWORD` | db | DB root. |
| `REDIS_PASSWORD` | misp-core, redis | Valkey/Redis AUTH. |
| `ADMIN_PASSWORD` | misp-core | MISP site-admin login. |
| `SALT` | misp-core | `security.salt`; set once, kept stable. |
| `GPG_PASSPHRASE` | misp-core | Passphrase locking the autogenerated instance GnuPG key. |
| `ADMIN_KEY` | misp-core, bootstrap | Fixed admin REST API key. |
| `SMARTHOST_PASSWORD` | misp-core | SMTP relay; empty in dev. |

The dotenv schema (with non-secret placeholders) is committed as
[`deploy/dev/secrets.example.env`](../deploy/dev/secrets.example.env).

> **Instance GnuPG key.** There is no committed key and nothing to seed. MISP
> autogenerates its instance GnuPG key on first boot, locks it with
> `GPG_PASSPHRASE`, and stores it in the `misp_gnupg` named volume. It is fresh
> per volume reset — `docker compose down -v` discards it and the next bring-up
> generates a new one. Dev does not pin this key (feed authenticity rides on
> mTLS + content hashes, not GPG-signature pinning).

## The inject path (single and obvious)

```
./bin/secrets-init.sh  ──generates──▶  secrets/misp.secrets.env  ──env_file:──▶  containers
                                         (git-ignored, 0600)                       (misp-core, db, redis)
```

`./bin/secrets-init.sh` generates the values; `docker-compose.yml` reads them
via `env_file:`. `./bin/up.sh` runs `secrets-init.sh` automatically on first
bring-up if the file is missing. Nothing secret is committed.

### Why `env_file:` and not Docker `secrets:`

Issue #3 names compose `secrets: { file: ./secrets/... }` as the example
mechanism. We use `env_file:` pointing at the same git-ignored `secrets/`
directory instead, for a concrete reason:

- Upstream `misp-core` reads its credentials **from environment variables**
  and has **no `_FILE` / `/run/secrets` convention**. Docker `secrets:`
  mounts files at `/run/secrets/<name>`, which the unmodified image never
  reads.
- Making the image consume `secrets:` would require **patching its
  entrypoint** — a fork-like change ruled out by the project invariant
  *"upstream MISP, no fork/build"* (issue #1).

So we keep the **invariant that matters** — nothing secret is committed and the
plaintext lives solely in the git-ignored `secrets/` dir — using the mechanism
the upstream image actually supports.

## Dev secret bootstrap procedure

### Prerequisites

Only `openssl` (used to generate the random values), which ships with macOS and
every Linux distro. No sops, no age, no GnuPG, no decryption key.

### Generate and bring up

```sh
cd deploy/dev
./bin/secrets-init.sh   # generate secrets/misp.secrets.env (or let up.sh do it)
./bin/up.sh             # starts the stack (generates secrets if missing)
./bin/bootstrap.sh      # idempotent fountel config (taxonomy, settings)
```

`up.sh` runs `secrets-init.sh` for you if `secrets/misp.secrets.env` does not
exist, mirroring how it seeds `.env` from `.env.example`. Running
`secrets-init.sh` first is optional.

`secrets/misp.secrets.env` (the generated plaintext, mode 0600) is created here
and is git-ignored. Never commit it.

### Re-running and rotation

- Re-running `./bin/secrets-init.sh` **without** `--rotate` is a **no-op**: it
  does not regenerate or clobber existing local secrets, so running it before
  every `up.sh` is safe and never invalidates a running instance.
- `./bin/secrets-init.sh --rotate` regenerates every value.

> **Rotation takes effect only after a volume reset.** MariaDB and MISP bake the
> DB password, `SALT`, and the GnuPG key into their persistent volumes
> (`mysql_data`, `misp_configs`, `misp_gnupg`) on first init and do not re-read
> them on a plain restart. The regenerated values will not match what is already
> in the volumes, so after `--rotate` you must reset and re-bring-up:
>
> ```sh
> docker compose down -v && ./bin/up.sh
> ```
>
> (This is also why all MISP state is kept in named volumes rather than host
> bind mounts — `down -v` resets DB, config, and the GnuPG key atomically, so
> they can never desync.)

## What is committed vs. ignored

| Committed | Git-ignored |
| --- | --- |
| `secrets.example.env` (placeholders, non-secret schema) | `secrets/` (generated plaintext, incl. `misp.secrets.env`) |
| `.env.example` (non-secret compose vars) | `.env` (local compose config) |

The `.gitignore` enforces this: `secrets/` and `*.secrets.env` are excluded.
No secret material — encrypted or otherwise — is committed.

## prod (Phase 2)

prod will inject secrets from a **secret manager**, not from anything in this
repo. That work is deferred to Phase 2 and tracked in #1's deferred list; it is
